üîç Security Researcher Workflow

Found a vulnerability in an open source project? Here's your step-by-step guide to responsible disclosure.

üìñ Comprehensive Guide Available: For the complete OpenSSF guide to coordinated vulnerability disclosure for security researchers, see the Finder's Guide to CVD and CVD Runbook.
1

Find the Security Policy & Contact

Before reporting, locate the project's preferred reporting method:

  • GitHub Security Policy: Look for SECURITY.md in the repository root or .github/SECURITY.md
  • Security Tab: Check the "Security" tab on GitHub repositories (if enabled)
  • Private Vulnerability Reporting: Some projects enable GitHub's private reporting feature
  • README.md: Check for security contact information or email addresses
  • Package Metadata: Look in package.json, setup.py, Cargo.toml, etc.
  • Bug Bounty Programs: Search HackerOne, Bugcrowd, or Intigriti for active programs

üí° Tip: If you can't find contact info, try ecosystem-level contacts for the language/platform, or use GitHub Security Advisories to report privately.

2

Assess Severity & Impact

Understand the severity to determine urgency and disclosure timeline:

üö® Critical (CVSS 9.0-10.0)
  • Remote Code Execution (RCE) without authentication
  • Active exploitation in the wild
  • Mass data exposure or credential leakage
  • Complete system compromise

Timeline: Report immediately, 7-14 day disclosure window

‚ö†Ô∏è High (CVSS 7.0-8.9)
  • Authentication bypass or privilege escalation
  • SQL injection, XSS in critical contexts
  • Significant data exposure
  • Crypto vulnerabilities affecting confidentiality

Timeline: 30-60 day disclosure window

‚ö° Medium (CVSS 4.0-6.9)
  • Information disclosure (non-critical)
  • DoS requiring significant resources
  • XSS in limited contexts
  • CSRF in non-critical operations

Timeline: 60-90 day disclosure window

‚ÑπÔ∏è Low (CVSS 0.1-3.9)
  • Security misconfigurations
  • Minor information leakage
  • Best practice violations
  • Theoretical vulnerabilities with no clear exploit

Timeline: 90+ day disclosure window

üí° Tip: Use the CVSS Calculator to determine the score objectively.

3

Prepare Your Report

Create a clear, actionable vulnerability report including:

Essential Information

  • Vulnerability Type: What category (e.g., RCE, XSS, SQLi, etc.)
  • Affected Versions: Which versions are vulnerable
  • Attack Vector: How can it be exploited (local/network/adjacent)
  • Prerequisites: What access level is needed
  • Impact: What can an attacker achieve

Reproduction Steps

  • Step-by-step instructions to reproduce
  • Environment details (OS, browser, dependencies)
  • Proof-of-concept code (if safe to share)
  • Screenshots or video demonstration (if helpful)
  • Expected vs actual behavior

Additional Context

  • Suggested Fix: Remediation recommendations (if you have them)
  • Related CVEs: Similar vulnerabilities in other projects
  • Discovery Method: How you found it (optional but helpful)
  • Disclosure Preferences: Your timeline and credit preferences

üí° Tip: Don't include exploits for active attacks or weaponized code. Share enough to validate, not enough to cause harm.

4

Report Securely

Choose the most secure communication method available:

‚úÖ Preferred Methods
  • GitHub Private Vulnerability Reporting: Built-in, encrypted, easy for both parties
  • Bug Bounty Platforms: HackerOne, Bugcrowd (managed process, potential rewards)
  • PGP-Encrypted Email: If maintainer provides a public key
  • Security-Specific Email: security@project.org (if listed in SECURITY.md)
‚ö†Ô∏è Acceptable (with caution)
  • Direct Email to Maintainers: For low-severity issues only
  • Encrypted Messaging: Signal, Wire (if maintainer provides contact)
‚ùå Avoid These
  • Public GitHub Issues (before coordinated disclosure)
  • Social media, forums, or public channels
  • Unencrypted email for critical vulnerabilities
  • Simultaneous public disclosure ("full disclosure")
5

Coordinate Disclosure

Work with maintainers for responsible disclosure:

  • Set Expectations: Propose a disclosure timeline based on severity
  • Be Patient: Many OSS maintainers are volunteers with limited time
  • Follow Up: If no response in 5-7 days, send a polite reminder
  • Escalate if Needed: After reasonable time, consider ecosystem coordinators or CVE Numbering Authorities
  • Agree on Disclosure Date: Coordinate public disclosure timing
  • Credit and Attribution: Discuss how you'll be credited (if desired)

Typical Timeline

  • Day 0: Initial report submitted
  • Day 1-7: Maintainer acknowledges receipt
  • Day 7-30: Maintainer confirms/validates vulnerability
  • Day 30-60: Fix development and testing
  • Day 60-90: Patched version released
  • Day 90+: Public disclosure and CVE publication

üí° Tip: If a maintainer is unresponsive or hostile, escalate to ecosystem coordinators rather than rushing to public disclosure.

üéØ Responsible Disclosure Best Practices

‚úÖ Do

  • Give maintainers reasonable time to fix
  • Communicate clearly and professionally
  • Provide actionable information
  • Respect embargo dates
  • Offer to help validate fixes
  • Document your findings thoroughly
  • Consider the impact on users

‚ùå Don't

  • Publicly disclose before coordination
  • Exploit vulnerabilities in production
  • Access more data than needed to prove impact
  • Threaten maintainers or demand payment
  • Share vulnerability details with others
  • Use automated scanning without permission
  • Rush to publish for credit or bounties

üõ°Ô∏è Preventive Security Best Practices

Implement security controls and best practices before vulnerabilities occur

Proactive security is more effective and less costly than reactive responses. These practices help you build security into your development process from the start.

üîí Code Security

Static Application Security Testing (SAST)

Analyze source code for security vulnerabilities before deployment

  • GitHub: CodeQL for automated code scanning
  • GitLab: Built-in SAST scanners for multiple languages
  • Open Source: Semgrep, Bandit (Python), ESLint plugins
Learn More

Secure Code Review

Establish security-focused code review processes

  • Review all code changes for security implications
  • Use security checklists and guidelines
  • Require multiple reviewers for sensitive code
  • Document security decisions and rationale
OWASP Code Review Guide

üì¶ Dependency Management

Dependency Scanning

Monitor and update dependencies with known vulnerabilities

  • Dependabot: Automated dependency updates for GitHub
  • Renovate: Multi-platform dependency automation
  • Snyk: Comprehensive vulnerability scanning
  • OWASP Dependency-Check: Free, open source scanner
Setup Dependabot

Software Bill of Materials (SBOM)

Maintain comprehensive inventory of software components

  • Generate SBOMs in standard formats (SPDX, CycloneDX)
  • Track all direct and transitive dependencies
  • Monitor for vulnerabilities in dependencies
  • Share SBOMs with users for transparency
SBOM Resources

üîê Access Control & Authentication

Repository Security

Protect your code repository and development infrastructure

  • Enable two-factor authentication (2FA) for all contributors
  • Use branch protection rules and required reviews
  • Limit write access to trusted maintainers only
  • Enable signed commits for verification
  • Rotate access tokens and credentials regularly
GitHub Security Settings

Secret Management

Prevent accidental exposure of sensitive credentials

  • Never commit secrets, API keys, or passwords to repositories
  • Use GitHub/GitLab secret scanning to detect leaks
  • Employ environment variables and secret managers
  • Rotate compromised credentials immediately
  • Use pre-commit hooks to prevent secret commits
Secret Scanning

üèóÔ∏è Build & Release Security

Supply Chain Security (SLSA)

Implement Supply-chain Levels for Software Artifacts framework

  • SLSA Level 1: Version control and build service documentation
  • SLSA Level 2: Hosted build service with provenance
  • SLSA Level 3: Hardened builds with non-falsifiable provenance
  • Use GitHub Actions or GitLab CI with SLSA builders
Implement SLSA

Artifact Signing & Verification

Cryptographically sign releases for authenticity verification

  • Sign releases with GPG or Sigstore/Cosign
  • Publish signatures and checksums with releases
  • Provide verification instructions for users
  • Use keyless signing with Sigstore when possible
Use Sigstore

üöÄ Getting Started - Quick Wins
